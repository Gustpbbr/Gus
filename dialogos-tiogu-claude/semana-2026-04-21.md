**Feito:**
- Verificação do código no arquivo `gus/llm.py` (linhas 203-209), função `gerar_resposta()`.
- Confirmado: **`datetime.now(BRT)` é chamado DENTRO de `gerar_resposta()`**, função que é invocada a cada mensagem que chega. Timestamp é gerado fresh a cada turno, não uma vez por sessão.
- Evidência:
  ```python
  async def gerar_resposta(messages, memory_context=""):
      ...
      system_prompt = _load_system_prompt()
      agora = datetime.now(BRT)            # ← fresh a cada chamada
      dia_semana = DIAS_SEMANA[agora.weekday()]
      system_prompt += f"\n\n## Data e hora atuais\nHoje é {dia_semana}, {agora.strftime('%d/%m/%Y')}, {agora.strftime('%H:%M')}..."
  ```
- O `_load_system_prompt()` tem `@functools.lru_cache` mas só cacheia a **leitura do arquivo** `system_prompt.md`, não o timestamp. A string final (prompt + timestamp) é reconstruída toda vez.

**Não feito:**
- Nenhuma mudança de código aplicada, porque o bug suposto não existe.

**Hipóteses pra data/horário errado (se acontecer de novo):**
1. **Fuso horário divergente** — `BRT = timezone(timedelta(hours=-3))` é fixo, não considera horário de verão (Brasil não usa DST desde 2019, então OK).
2. **Railway sem redeploy** — se `system_prompt.md` for alterado e Railway não redeployar (raro), o bot usa prompt cacheado. Mas o timestamp ainda seria fresh. Esse cache é só texto.
3. **Gus especulando em vez de verificar** — o bot aceitou a hipótese do Gustavo sem consultar o próprio código via `read_from_github("gus/llm.py")`. Quando houver ambiguidade sobre o próprio funcionamento, a ferramenta já existe pra verificar.
